# v2.1.0 开发计划（TDD）

> 依据：`docs/comprehensive_repost.md` 审计项 D-2026-06 + `docs/ROADMAP.md` v2.1.0 R-1/R-2/R-3。
> 目标周期：7~10 工作日（本次交付为一次性重构落地）。

## 1. 范围与验收口径

### 1.1 交付范围
- R-1：`reader.js` 从单文件拆分为 `reader-runtime.js`、`reader-state.js`、`reader-persistence.js`、`reader-ui.js`，入口文件仅保留编排逻辑。
- R-2：建立统一生命周期 `mount(context)` / `unmount()`，由入口统一挂载 Reader 子模块。
- R-3：移除 Reader 内部跨作用域隐式状态共享，改为显式 `state/context` 传递。
- 清理：删除 `DbGateway.getByFilename()` 死代码（D-2026-06）。

### 1.2 验收标准（可自动化验证）
1. `reader.js` 行数 < 120。
2. 四个新 Reader 子文件行数均 < 250。
3. 各分层文件均包含 `mount/unmount` 生命周期契约。
4. `reader.html` 脚本加载顺序包含 4 个新模块并先于 `reader.js`。
5. `DbGateway.getByFilename()` 在源码中不存在。
6. 版本号升级到 `2.1.0`，文档同步更新。

## 2. TDD 实施步骤

### 阶段 A：先写测试（Red）
- 新增 `test/suites/v2_1_tdd.test.js`，覆盖：
  - 文件拆分与行数门禁。
  - 生命周期接口门禁。
  - 入口编排门禁（ReaderState/Runtime/UI/Persistence 协作）。
  - `db-gateway.js` 死代码删除门禁。
  - `manifest.json` 版本号门禁。
- 更新 `test/suites/v2_0_tdd.test.js` 与 `test/suites/release_checks.test.js` 中与 2.0 强绑定的版本断言，提升为 2.1 基线。

### 阶段 B：最小实现（Green）
- 新建 Reader 四层文件并实现最小可运行能力。
- 重写 `reader.js` 为单一入口 orchestrator。
- 更新 `reader.html` 脚本依赖顺序。
- 删除 `DbGateway.getByFilename()`。

### 阶段 C：回归与修正（Refactor）
- 运行 `node --test test/suites/*.test.js`。
- 运行 `node test/run_tests.js` 做历史回归。
- 修正失败项并重跑至全绿。

## 3. 风险与回滚
- 风险1：Reader 初始化流程变更导致模块挂载时序回归。
  - 缓解：入口维持原模块初始化顺序（ImageViewer → Annotations → TOC → Search → Bookmarks → Highlights）。
- 风险2：拆分后上下文漏传。
  - 缓解：统一 `context` 字段 `{ book, rendition, bookId, fileName }`。
- 回滚策略：保留独立提交粒度，必要时可回滚到拆分前版本。
