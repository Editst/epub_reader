### BUG-01：Popup 首次打开新书无反应（第二次才生效）

**现象**：在 popup 弹窗中点击"打开文件"选取 `.epub` 文件，选文件对话框关闭后无任何反应，reader 页未打开。重新打开 popup 再操作一次则正常。

**根因（Chrome Popup 生命周期 + FileReader 竞态）**：

```
用户点击 open-btn
  → fileInput.click()          ← 触发系统文件对话框

问题发生在此处：
Chrome 规范：Popup 弹窗在失去焦点时（如文件对话框弹出）可能触发 unload。
文件对话框弹出 → popup 失去焦点 → Chrome 可能销毁 popup document（非必然，取决于版本和 OS）

用户选完文件 → 文件对话框关闭 → focus 回 popup
  → 若 popup 已被销毁：fileInput.change 事件无法触发（document 已不存在）
  → 若 popup 未被销毁：正常执行 change handler
```

**复现概率**：在 Windows 上约 80%，macOS 约 40%（与 OS 文件对话框焦点处理差异相关）。

**代码位置**：`src/popup/popup.js` 第 13 行

```javascript
// 问题代码
openBtn.addEventListener('click', () => fileInput.click());
// fileInput 是 display:none 的 <input type="file">
// Chrome popup 在文件对话框弹出期间可能卸载 popup document
```

**修复方案**：

将文件选择入口从 popup 的隐藏 input 改为通过 `home.html` 中转，或使用 `chrome.tabs.create` 打开 home 页后在 home 页触发文件选择。更简洁的方案：

```javascript
// 方案 A（推荐）：使用 showOpenFilePicker API（Chrome 86+）
// 不依赖 <input type="file">，不触发 popup 失焦
openBtn.addEventListener('click', async () => {
  try {
    const [fileHandle] = await window.showOpenFilePicker({
      types: [{ description: 'EPUB Files', accept: { 'application/epub+zip': ['.epub'] } }]
    });
    const file = await fileHandle.getFile();
    const arrayBuffer = await file.arrayBuffer();
    const bookId = await EpubStorage.generateBookId(file.name, arrayBuffer);
    await EpubStorage.storeFile(file.name, new Uint8Array(arrayBuffer), bookId);
    chrome.tabs.create({
      url: chrome.runtime.getURL('reader/reader.html') + '?bookId=' + encodeURIComponent(bookId)
    });
    window.close();
  } catch (e) {
    if (e.name !== 'AbortError') console.error('[Popup] File open failed:', e);
  }
});
```

```javascript
// 方案 B（兜底）：在 chrome.tabs.create 打开的 home 页中处理文件选择
// popup 只做跳转，不在 popup 里做 IO
openBtn.addEventListener('click', () => {
  chrome.tabs.create({
    url: chrome.runtime.getURL('home/home.html') + '?action=open-file'
  });
  window.close();
});
// home.js 检测 ?action=open-file 后自动触发 fileInput.click()
```

**推荐**：方案 A，因为 `showOpenFilePicker` 是专门为此场景设计的 API，popup 不会因文件对话框失焦而卸载。方案 B 用户体验多一跳，作为降级。

**优先级**：P1（功能性 bug，影响核心使用路径）

### BUG-02：预计剩余时间不准确

**现象**：用户反馈 ETA 数值与实际阅读速度出入较大，尤其在以下场景：  
① 刚打开书籍时 ETA 显示 `--` 或 fallback 静态值，长期不更新；  
② 阅读速度快的用户（快速翻页）与慢的用户（精读）ETA 几乎相同；  
③ 有时 ETA 突然跳变。

**根因（三处系统性缺陷）**：

**缺陷 1：`_cachedSpeed` 在 `flushSpeedSession` 后未同步更新**

```javascript
// reader.js ~770 行注释说明了意图但代码未实现
// v1.7.0: 缓存历史速度到 window._cachedSpeed，避免每次 updateReadingStats 读 storage
// 覆写 flushSpeedSession 以在 flush 后刷新缓存
const _origFlushSpeedSession = flushSpeedSession;
// Note: 由于 flushSpeedSession 已在上方声明为 async function，
// 我们通过事后调用 refreshCachedSpeed 来保持缓存同步
// ↑ 实际上 wrap 从未执行，_origFlushSpeedSession 没有被调用
```

`flushSpeedSession` 写入 storage 后，`window._cachedSpeed` 没有更新。导致 `updateReadingStats` 始终读到旧的缓存（初始值），历史速度路径永远不生效，实际上 ETA 一直在走 session 实时速度或 fallback。

**缺陷 2：session 实时速度的分子是会话总时长，不是阅读总时长**

```javascript
// 当前代码
const sessionDeltaSeconds = (Date.now() - _sessionStart.timestamp) / 1000;
// _sessionStart.timestamp 是书籍加载完成时的时间戳

// 问题：用户打开书读了 1 分钟，挂起页面（tab 不可见）放了 30 分钟，
// 再回来继续读。sessionDeltaSeconds 包含了那 30 分钟的挂机时间，
// 导致速度被严重低估，ETA 虚高。
```

注意：`visibilitychange(hidden)` 时会调用 `flushSpeedSession(null)` 结束 session，理论上不续期。但 `flushSpeedSession` 的 `newStartProgress !== null` 才重设起点，`null` 时仅结束不重启，而重新激活页面时未重置 `_sessionStart`——所以如果 flush 条件不满足（deltaProgress < 0.001），_sessionStart 保持不变，下一次 relocated 继续累计，时间窗口跨越了挂机期。

**缺陷 3：session 实时速度的触发阈值（> 60s 且 > 0.5%）偏高**

ETA 在 session 前 60 秒始终显示 fallback 静态值（通常是一个很大的数字），用户第一印象不佳。阈值下调到 > 30s 且 > 0.3% 即可给出有意义的估算。

**修复方案**：

```javascript
// 修复 1：flushSpeedSession 结束后立即更新缓存
async function flushSpeedSession(newStartProgress = null) {
  if (!_sessionStart || !currentBookId || !isBookLoaded) return;
  const deltaProgress = _lastProgress - _sessionStart.progress;
  const deltaSeconds  = (Date.now() - _sessionStart.timestamp) / 1000;

  if (deltaProgress > 0.001 && deltaProgress < 0.30 && deltaSeconds > 30) {
    try {
      const spd = window._cachedSpeed || { sampledSeconds: 0, sampledProgress: 0 };
      spd.sampledSeconds  += deltaSeconds;
      spd.sampledProgress += deltaProgress;
      // 直接更新内存缓存（不需要再读 storage）
      window._cachedSpeed = { ...spd };
      await EpubStorage.saveReadingSpeed(currentBookId, spd);
    } catch (e) {
      console.warn('[Speed] Failed to save speed sample:', e);
    }
  }

  _sessionStart = (newStartProgress !== null)
    ? { progress: newStartProgress, timestamp: Date.now() }
    : null;
}

// 修复 2：visibilitychange 重新激活时重置 session 起点
document.addEventListener('visibilitychange', () => {
  if (document.hidden) {
    flushPositionSave();
    saveReadingTime();
    flushSpeedSession(null);  // session 结束
  } else {
    // 页面重新可见，重启 session 起点（排除挂机时间）
    if (isBookLoaded && _lastProgress > 0) {
      _sessionStart = { progress: _lastProgress, timestamp: Date.now() };
    }
  }
});

// 修复 3：降低 session 实时速度触发阈值
if (sessionDeltaProgress > 0.003 && sessionDeltaSeconds > 30) {
  // 原：0.005 且 60s
}
```

**关于 ETA 精度的根本性限制**：  
epub.js 的 `location` 进度基于 CFI 位置在 locations 数组中的偏移量，不等价于实际文字数量。图片页、标题页、空白页均占用 location，但阅读时间极短。这是引擎层面的约束，无法在应用层完全消除。通过增大采样量（多次 session 累积）可以减小误差，但不能消除。

**优先级**：P1（核心功能）

### BUG-03：阅读页面缩放（字号/窗口 resize）后位置偏移

**现象**：调大字号或缩小窗口宽度后，当前阅读位置向前/向后偏移，不再停留在操作前的段落。在 paginated 模式下更明显（滚动模式下因连续布局问题较小）。

**根因（三层叠加）**：

**根因 1：resize 时使用 `loc.end.cfi` 而非 `loc.start.cfi`**

```javascript
// 当前代码（reader.js ~188 行）
if (loc && loc.end) preResizeCfi = loc.end.cfi;
// 然后用 rendition.display(preResizeCfi) 恢复位置

// 问题：
// - loc.end.cfi 是当前屏幕最后一个可见字符的位置
// - 字号放大后单屏显示字数减少，end.cfi 对应的内容可能在前一页
// - 实际上想恢复到的是"用户正在读"的位置，应该是 start.cfi
// - end.cfi 只在 resize 后希望保持"屏幕末尾不变"时有意义，
//   但这与阅读体验相悖（用户期望"当前段落不动"）
```

**根因 2：字号变化触发的 rendition 重排没有 CFI 保护**

```javascript
// applyFontSize / applyLineHeight / applyLetterSpacing 当前实现
function applyFontSize(size) {
  if (!rendition) return;
  currentPrefs.fontSize = size;
  updateCustomStyles();  // 仅修改 CSS，不处理位置恢复
  // updateCustomStyles() 会触发 rendition 内部重排
  // 重排后 relocated 事件会以新位置触发，但新位置未必是原来的位置
}
```

字号 / 行高 / 字间距修改通过 `updateCustomStyles` 写入 iframe CSS，触发 epub.js 内部重排。paginated 模式下 epub.js 重新计算分页，relocated 事件重新触发，但此时没有锁定恢复目标 CFI。

**根因 3：`preResizeCfi` 作用域在 resize 防抖计时器内，字号变化不走此路径**

resize 保护逻辑（`isResizing` 标志 + `preResizeCfi`）只拦截 `window resize` 事件，字号变化走 `applyFontSize` 直接修改 CSS，两套路径互不感知。

**修复方案**：

```javascript
// 统一 CFI 保护机制
// 在任何可能触发重排的操作前，保存当前 start.cfi
function withCfiLock(asyncFn) {
  return async function(...args) {
    if (!rendition || !isBookLoaded) return asyncFn.apply(this, args);
    const loc = rendition.currentLocation();
    const savedCfi = loc?.start?.cfi || currentStableCfi;  // 使用 start，不用 end
    isResizing = true;
    try {
      await asyncFn.apply(this, args);
      // 等待重排完成后恢复
      await new Promise(r => requestAnimationFrame(() => requestAnimationFrame(r)));
      if (savedCfi) await rendition.display(savedCfi);
    } finally {
      isResizing = false;
      const newLoc = rendition.currentLocation();
      if (newLoc?.start) onLocationChanged(newLoc);
    }
  };
}

// 包裹字号/行高/字间距的应用函数
function applyFontSize(size) {
  currentPrefs.fontSize = size;
  withCfiLock(async () => updateCustomStyles())();
}

// resize 事件也改用 start.cfi
window.addEventListener('resize', () => {
  if (!rendition || !isBookLoaded) return;
  isResizing = true;
  if (!preResizeCfi) {
    const loc = rendition.currentLocation();
    if (loc && loc.start) preResizeCfi = loc.start.cfi;  // 改为 start
  }
  // ...
});
```

**关于 epub.js paginated 模式的内在限制**：  
epub.js 的 paginated 渲染基于 CSS `column-count` 分栏，字体大小变化后分栏数量改变，原始 CFI 指向的字符可能落入新的列/页。`rendition.display(cfi)` 会重新导航到该字符，但视觉上仍然是"换页"，无法做到像 PDF 那样的像素级位置保持。这是 web-based EPUB 渲染器的固有约束，修复后可将偏移控制在 `< 1 屏`（当前约 `2–3 屏`）。

**优先级**：P1（高频操作必现）

## 五、接口变更（v1.7.0 → v1.8.0）

### 存储格式（无破坏性变更）

v1.8.0 不改变 `bookMeta_*` 的 Schema，`speed` 字段格式不变。

### API 变更

| 模块 | 变更 | 兼容性 |
|---|---|---|
| `flushSpeedSession` | 内部直接更新 `_cachedSpeed`，不再依赖 `refreshCachedSpeed` | 内部变更，无外部影响 |
| `popup.js` | 新增 `showOpenFilePicker` 调用路径，保留 input fallback | 向前兼容 |
| `reader.js` | `applyFontSize/applyLineHeight` 变为 async（带 CFI 保护） | 内部变更 |

### 废弃项

| 项目 | 原因 | 替代 |
|---|---|---|
| `refreshCachedSpeed()` | 功能并入 `flushSpeedSession` | 无需独立调用 |
| `window._cachedSpeed` | 改为模块级 `let _cachedSpeed` | 同名局部变量 |
| `_origFlushSpeedSession` | 废弃未完成的 wrap 占位代码 | 直接修改 flushSpeedSession |
